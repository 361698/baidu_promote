# ku-doc-manage experiment

## 特别提示：最容易错的地方

- 新建或完整写回 KU 正文时，`create-doc --content` / `create-doc --md-file` / Markdown 原文写入都不是成功路径。稳定路径只有：先 `create-doc --create-mode empty`，再把正文转换成 KU 编辑器 JSON，用 `edit-content --editor-mode cover` 覆盖正文，发布后读回。
- 首轮创建后如果读回发现正文只剩标题、正文上方有空卡片、Markdown 被收进标题节点或隐藏内容里，都是失败形态，不是正常结果。立即用编辑器 JSON `cover` 覆盖正文，并用 `query-content --protocol json` 检查真实正文节点。
- 写作必须围绕用户原问题收束。用户只要求“总结/梳理/提炼”时，不要擅自新增“建议、后续动作、待确认问题、风险、方案设计”等章节；除非用户明确要建议，或材料本身必须用“待确认”标注事实缺口。
- 写入前先做结构门禁：顶级章节原则上 2-4 个；出现“五、六、七、八、九”这类长编号，或把建议后续动作、证据摘要、参考来源、待对齐拆成多个顶级章，先压缩重组，不要写入。
- 善用表格。时间线、方案对比、最终口径、行动清单、证据摘要这类信息，如果不用表格会变成长段并列文字，默认先设计成表格；没有表格通常要先解释为什么不用表格。
- 群聊、会议、文档来源要使用可识别名称，不要写“群1/群2/12917973群聊”这类只有抓取者知道的编号；如果只拿到编号，先查或标注“群名待确认”，不要把编号当正式名称。
- 发布前必须复核：正文 JSON 不是空卡片，标题节点没有混入正文 payload，表格是 `table/table-row/table-cell`，顶级章节未超限，应表格化的信息已经表格化，再发布。

## 定位

`ku-doc-manage` 负责 KU/如流知识库文档的读取、创建、编辑、发布、评论、权限、附件、在线表格和数据表操作。本文沉淀可复用的传参、写作、编辑、验证经验。

## 传参要求

- KU 普通文档 URL 最后一段是 `docGuid`，倒数第二段是 `repositoryGuid`。
- 每条 KU 命令都要在实际子进程环境里设置 `SANDBOX_USERNAME=<uuap>`；只传 `--username` 不够，包装器仍可能认证失败。
- 不要只看 shell 退出码；KU API 错误可能仍退出码为 0，要检查输出 JSON 的 `success`、`status`、`returnCode`。
- `query-content --protocol markdown` 的正文通常在 `result.text`；`query-content --protocol json` 的正文通常在 `result.content`。
- 评论不在正文 JSON 中，必须用 `query-comments` 单独读取。
- `create-doc --content` / `create-doc --md-file` 不是稳定正文写入路径；稳定创建正文流程是 `create-doc --create-mode empty` -> 把正文转换成 KU 编辑器 JSON 节点 -> `edit-content --editor-mode cover` -> `publish-doc` -> `query-content` 读回。若 Markdown 进入标题隐藏内容、正文只剩标题或空卡片，视为写入失败，必须 cover 重写正文。
- `edit-content` 后必须 `publish-doc`，否则修改可能只停留在草稿。
- `append` 只用于已有文档的新增小节、追加总结、追加图片等小范围增量；新建文档首条正文、完整正式文档写回、替换、删除、中间插入、修改表格行必须读/构造 JSON 后定位并 `cover`。
- `cover` 前保存备份；只修改目标节点，保留无关节点。`result.content[0]` 常是标题节点，覆盖正文时不要写进正文 payload，避免标题重复。
- 图片先 `upload-attachment` 到目标文档，再用目标文档自己的 `attachId` 生成图片地址：`https://rte.weiyun.baidu.com/wiki/attach/image/api/imageDownloadAddress?attachId=<attachId>&docGuid=<docGuid>`。
- Markdown 是读回/草稿格式，不是正式写回格式。正式文档必须把标题、段落、列表、表格、代码块、图片转换成 KU 编辑器 JSON 节点；不要把 Markdown 原文塞进 `paragraph` 或 `block-code` 当正文。
- 表格要写 KU `table` 节点，不要退化成代码块，也不要写成 Markdown 管道符文本。`table.data.width` 控制列宽，表头灰底要给首行每个 `table-cell.data.backgroundColor` 写 `rgb(231, 230, 230)`。

## 内容要求

- 用户说“用第一性原理”时，指的是写之前先用第一性原理拆解问题——回到目标本身、剥掉惯例和模板、从最基本的事实和约束重新推导结论，再决定文档怎么写。这是思考方式，不是文档内容；不要把“第一性原理”这几个字写进正文，也不要硬加“第一性原理分析”之类的小节标题。
- 第一性原理落到 KU 文档上，好的样子是：先抓出真正驱动结论的根本约束（例如“不能亏 + 必须卖得动”这类彼此拉扯的硬约束），再把结论拆成最基本的构成量（如成本=单价×消耗、占比=成本/售价），用基本事实和测算一步步推导，每个结论都能回溯到具体数字或事实，而不是给“周期波动”“要有弹性”这类无法判断的标签。呈现上可以把“怎么一步步演化到现在/为什么这么定”和“最终口径”分开写，最后用一句话回到根本逻辑收口。
- 反面（线上那种不够第一性原理的写法）：只是把已有内容重新排版、换措辞、按模板补章节，没有回到约束重新推导，也没有把数字拆开算清。识别到这种倾向就停下来，先想清根本约束和推导链，再动笔。
- 先回答用户问的那个问题，再决定是否需要扩展。用户只说“总结”就只总结；用户要“问题清单”才列待确认；用户要“建议”才写建议。不要为了显得完整而主动加行动项、风险、待确认、参考来源，除非缺少这些会影响用户判断。
- 只写能帮助决策的信息，不要为了像 PRD 而堆章节。信息量不够时，保留背景、目标、方案、关键规则、待讨论即可。
- 章节宁少勿多，绝不要写得冗长。动笔前先定位“用户真正要回答的那一个核心问题”，围绕它搭一条清晰故事线（通常一条主线 + 一个最终结论/计划就够），不要把内容拆成一堆并列章节平铺。判断标准：超过 3-4 个顶级章节，或出现“核心结论 / 时间线 / 方案变化 / 最新计划 / 待对齐 / 落地顺序 / 参考来源”这种七段平铺、各章互不咬合时，就是太散，要合并重组。
  - 好的样子（参考“变化主线 + 最终计划”两段式）：用一张时间线/主线表把演化讲清，再给最终口径表和“为什么”，最后一句话收口；读者能一眼抓到主问题。
  - 坏的样子：面面俱到地把所有方面各起一节、各配子节，看着全面但没有故事线，读者抓不到主问题——这不是详尽，是冗长。
- 写入前先做结构门禁：先列拟写顶级章节，再压缩。顶级章节原则上控制在 2-4 个；超过 4 个必须是用户明确要求或材料天然分层，否则合并。不要把建议后续动作、证据摘要、参考来源、待对齐分别拆成独立顶级章；它们应并入主线、最终口径、待讨论或附注。
- 如果草稿已经出现“七、八、九”这类长编号，停止写入，先重组为“主线表 + 最终口径 + 待讨论/证据附注”一类更短结构。
- 优先用表格压缩信息（时间线、对比、最终口径、行动清单、证据摘要），代替多个并列小节的反复展开。能用一张表讲清的变化，不要拆成三四节文字；写完没有任何表格时，主动检查是否漏掉了本该表格化的信息。
- 宁可砍掉与核心问题无关的并列章节，也不要为求全面写成倾倒式 PRD。待对齐、落地顺序这类附属内容，能并进主线或压成一小段就不要单列成章。
- 分清“已确定”和“待讨论”。用户明确给出的方案、约束、口径写进正文方案区；没定的内容放入“待讨论”，不要替用户拍板。
- 结论先行，但结论要是可判断的句子，不要只是标签。例如“按月清零会浪费低峰、卡住高峰”比“周期波动”更有用。
- 展开句只解释当前结论，不另起新观点；避免一段里塞多个判断。
- 表格列数服务决策问题，能三列讲清楚就不要扩成六列。用户要求压成三列时，通常是在提醒合并辅助说明。
- 涉及分位数、日均值、覆盖天数、渗透率、风险阈值时，先定义口径，再放推算；口径表和推算表分开更清楚。
- 用产品文档语气：直接、可判断、不过度口语。推荐“权益清晰”“价格策略要有弹性”“高频风险要可控”；避免“买到了什么”“快速救急”“别人权益”。
- 用户说“太通俗”时，优先把口语词换成产品词；用户说“太死板”时，优先把抽象标签换成判断句。
- 不确定的事实写“待确认”，不要把推测写成事实。
- 不要补用户没有给过的竞品、指标、风险、流程；除非它确实支持判断，且能标注为待确认。
- 引用群聊、会议或资料来源时，用群名、会议名、文档名等可识别名称。不要用纯序号或内部抓取编号代替名称；确实只有编号时，先补查名称，查不到再写“群名待确认（编号：xxx）”。

## 呈现要求

- 表格列宽按每列内容的平均字数占比分配；文本多的列更宽，文本少的列更窄，但每列都要保留可读的最低宽度，不要使用固定总宽或固定魔法数。
- Markdown 表格在本地撰稿时每行列数必须一致；写入 KU 前必须转换成 `table/table-row/table-cell` 节点。单元格压缩只用于“与上一行相同”的情况：短语、元素、原因、输入类列可写 `-`，属性、说明、结果、影响等论述类列可写 `同上`；真实缺失的信息按语义写明“待补充”“无”或留给用户确认，避免行尾紧贴 `||`。
- 不要出现 `备注 | 备注 | ...` 这种无信息重复列，改成 `概要`、`规则`、`影响`、`处理方式` 等具体字段。
- 中文段落不要用多个空格制造视觉间隔；用标点、换行、列表或表格列。
- HTML demo 需要进 KU 时，先截图成 PNG 再插入 image；HTML、Excel、PDF、ZIP 更适合附件卡片。
- Mermaid 必须写成 KU 编辑器 JSON 的 `block-code` 节点，`data.language` 设为 `"mermaid"`，源码放在 `children[0].text`、多行用真实换行；不要用 `diagram` 节点，也不要写成普通 Markdown 的 ```mermaid 围栏文本（围栏在 KU 里只是纯文本，不渲染）。读回时 KU 可能把代码归一化成 `block-code-line` 子节点，识别时两种形态都要兼容。
- 选视图：只要源码就放一个 `block-code`；只要效果图就放一张 PNG `image`（先把图渲染好上传到目标文档）；要“源码 + 效果图”就两块成对，先 `block-code` 再紧跟同一份源码渲染出的 PNG，图片 `src` 里的 `docGuid` 必须是目标文档自己的 ID。`edit-content --editor-mode cover` 的 `json` 只传正文节点数组，不传 `title` / `card` / `card-item` 包装层。发布后读回 JSON 核对 Mermaid 代码块、图片节点和图片 `docGuid`。

## 验证与沉淀

- 写完必须读回 Markdown 和 JSON：确认发布成功、正文可见、正文不只是标题或空卡片、Markdown 没有进入标题隐藏内容、表格是 `table/table-row/table-cell`、图片使用目标文档 `attachId`。
- 发布前检查标题层级和表格使用；如果顶级章节超过 4 个且用户没有明确要求长文，或时间线/对比/行动清单/证据摘要没有表格，先压缩重写，不要发布。
- 替换类任务要确认旧词消失、新词出现在原位置；不要用追加“生效口径”小节糊过去。
- 用户纠正结构、语气、表格、范围或判断方式时，判断是否可复用；可复用则更新本文，单次业务事实不要沉淀。

## 安全边界

- 不把 UGate token、Bearer token、AK/SK、真实业务链接写入本文。
- 示例用 mock 占位。
- 删除文档、移动文档、修改权限前必须二次确认。
